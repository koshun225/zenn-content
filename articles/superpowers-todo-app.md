## プロンプトひとつで、ここまでできた

最初に投げたプロンプトは、これだけです。

```
TODOアプリを作りたい。superpowersの使い方を学ぶチュートリアル的に作ってみたい。
その内容をZennの記事にしたいから、そこそこ見栄えがいいやつで。
ただし実装はシンプルに。pythonのfastapiを使って。フロントエンドはお任せで。
```

そして、できあがったのがこちら。

![完成したTODOアプリ](/images/superpowers-todo-app/image.png)

FastAPI + HTMX + SQLiteで動くTODOアプリ。タスクの追加・完了・編集・削除・フィルターまでついて、デザインもNotion風のミニマルな仕上がりです。

このアプリは、上のプロンプトから始まって、Claudeの質問に答えていくだけで完成しました。コードを1行も自分で書いていません。でも、技術選定もデザインの方向性も、すべて自分で決めた実感があります。

その体験を支えていたのが **Superpowers** というClaude Codeのプラグインでした。

## Superpowersとは

Superpowersは、Claude Codeに「設計フェーズ」を追加するプラグインです。インストールは1行で終わります。

```
claude install-plugin superpowers
```

導入すると、`brainstorm`で要件を対話的に固め、`plan`で実装手順を整理し、コードを書いたら`code-review`で品質をチェックする——という一連のワークフローが使えるようになります。

ポイントは、「AIにコードを出してもらう」のではなく、「AIと対話しながら設計し、自分で判断して進める」ということです。

## brainstormが曖昧なプロンプトを仕様に変えた

### プロンプトと仕様書の比較

まず結果から見てください。自分が書いたプロンプトと、brainstormを経て出来上がった仕様書を並べてみます。

| 項目 | プロンプトで言ったこと | 仕様書で決まったこと |
|------|----------------------|---------------------|
| Backend | FastAPI | FastAPI + SQLAlchemy + SQLite |
| Frontend | 「お任せ」 | HTMX + Jinja2（Node.js不要） |
| デザイン | 「見栄えがいい」 | ミニマル・モノトーン（Notion風） |
| 構成 | 「シンプルに」 | main.py 1ファイルに集約 |
| 機能 | 言及なし | CRUD + フィルター + インライン編集 |
| テスト | 言及なし | pytest + in-memory SQLite |

「お任せ」「シンプルに」「見栄えがいい」——こうした曖昧な指示が、具体的な技術選定と設計方針に変換されています。しかもこれは勝手に決められたわけではなく、brainstormでの問答を通じて自分で選んだ結果です。

### brainstormでの問答の流れ

では、実際にどんなやりとりがあったのか。おおよそこのような問答がありました。

**Q1. フロントエンド技術はどうしますか？**
プロンプトでは「お任せ」と書いただけでしたが、Claudeは候補を提示してきました。React、Vue.js、HTMX + Jinja2の3択です。「npmのセットアップが不要で、チュートリアルとしてシンプルに保てる」という理由からHTMX + Jinja2が推奨されていて、それを選びました。

**Q2. 機能のスコープはどこまで？**
TODOアプリといっても幅があります。基本のCRUD（追加・表示・編集・削除）に加えて、完了/未完了のフィルター機能を入れるかどうか聞かれました。チュートリアルとしてちょうどいい粒度だったので、フィルターも含めることにしました。

**Q3. デザインの方向性は？**
「見栄えがいい」だけでは作れないので、具体的なデザイン案を3つ提示してくれました。それぞれビジュアル付きで比較でき、その中からミニマル・モノトーンのNotion風を選びました。

![デザイン選択画面](/images/superpowers-todo-app/作業中.png)

**Q4. プロジェクト構成はどうしますか？**
ファイルを分割してしっかり構成するか、`main.py`にまとめるか。「シンプルに」と言っていたので、シングルファイル構成が提案されました。チュートリアル用途なのでそれで十分です。

**Q5. テストはどうしますか？**
テストについてはプロンプトで何も触れていませんでしたが、Claudeから「テストも入れておきますか？」と提案がありました。in-memoryのSQLiteを使えば本番DBに影響せずテストできるとのことで、入れることにしました。

### 質問の「質」が違う

ここで少し立ち止まって、brainstormの質問の仕方について触れたいと思います。

v0やCursorなど他のAIツールは、プロンプトを入れるとすぐにコードを生成します。それはそれで便利ですが、「自分が何を作りたかったのか」が曖昧なまま進んでしまうことがあります。

Superpowersのbrainstormは違います。いきなりコードを書くのではなく、まず理解しようとします。そして質問は一度に1つずつ。矢継ぎ早に聞いてくることはありません。

しかも、質問は選択肢形式で提示されます。たとえばフロントエンドの質問はこんな感じでした。

> **フロントエンドの技術はどうしますか？**
> - A. React — リッチなUIが作れるが、npmセットアップが必要
> - B. HTMX + Jinja2（推奨）— npm不要でシンプル、チュートリアル向き
> - C. Vue.js — Reactより軽量だが、それでもビルドツールが必要

各選択肢に「プロジェクトにどう影響するか」の説明がついています。そして、どれが推奨かも明示されています。

これが大きいと感じました。技術判断に自信がない領域でも、影響範囲を理解した上で選べます。「ReactとHTMXの違いはよくわからないけど、npmが不要でシンプルなほうがチュートリアルには合っている」——そういう判断ができるわけです。

結果として、「AIが勝手に決めた」ではなく「自分で選んだ」という納得感が残ります。これはコードの品質とは別の、でもとても重要な体験の違いです。

## brainstormの先——plan・実装・code-review

brainstormで「何を作るか」が決まったら、次は「どう作るか」です。

### plan：実装ステップの自動生成

brainstormの成果物である仕様書をもとに、planが実装手順を自動で組み立てます。たとえば今回のTODOアプリでは、こんな順番が提案されました。

1. プロジェクトのセットアップ（FastAPI + SQLAlchemy + SQLite）
2. ベーステンプレートとGETルートの実装
3. タスク追加のPOSTエンドポイント
4. 完了トグルと削除のエンドポイント
5. 編集とフィルターのエンドポイント
6. CSSスタイリング
7. テストの追加

何をどの順番で作るかが整理されるので、途中で迷子になることがありません。実際のgitログを見ると、このステップ通りにコミットが積まれていて、計画がそのまま実行されたことがわかります。

### 実装：計画に沿ったコード生成

planで決めた順番に従って、Claudeがコードを書いていきます。ここは他のAIコード生成と似ていますが、違うのは「何を作るか」がbrainstormで明確になっているので、出てくるコードにブレがないことです。

### code-review：自動品質チェック

各ステップの完了時に、code-reviewerが自動でレビューを実行します。計画通りに実装されているか、品質に問題がないかをチェックしてくれます。

人間のコードレビューと同じで、問題があれば指摘が入ります。自分でレビューしなくていいというわけではありませんが、明らかなミスは事前に潰してくれるので安心感があります。

### 一貫したワークフロー

brainstormで「何を作るか」を決め、planで「どう作るか」を整理し、code-reviewで品質を担保する。この3つが一貫したワークフローとして自動的に回ることが、Superpowersの本質だと感じました。

個々の機能はそこまで革新的ではないかもしれません。でも、設計から実装、レビューまでが途切れなくつながっている体験は、他のツールにはないものでした。

### Superpowersを使ってみて感じたこと

正直、最初は「質問に答えるだけで本当に大丈夫か？」と思っていました。でもbrainstormが終わった時点で、作りたいものの全体像がはっきり見えていた。あとはplanに沿って進めるだけでした。

自分はフロントエンドにそこまで詳しくないので、HTMXを選んだ判断が正しかったのかはまだわかりません。でも「なぜHTMXにしたのか」を自分の言葉で説明できる。それだけで、AIが生成したコードとの付き合い方が変わると感じました。
